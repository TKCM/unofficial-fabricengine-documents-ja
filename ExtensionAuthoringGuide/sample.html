Understanding the Sample Extension¶

In this section we go through each of the files in the sample extension that you built in the last section. By understanding how these files work and fit together you will understand the basics of how extensions work in Fabric Engine.

HelloWorld.fpm.json¶

The only required file in an extension is the extension manifest file; in the sample extension it is the file HelloWorld.fpm.json, and its contents are:

{
  "libs": "HelloWorld",
  "code": "HelloWorld.kl"
}

The extension manifest file must end with the .fpm.json suffix, and the part of the filename leading up to .fpm.json is by definition the name of the extension. The name of the extension is what is used include the extension using the require ExtensionName; statement in KL or using the exts parameter when creating a Fabric Engine core client.

The contents of the manifest file are in JSON format (see http://json.org/), and must be a single JSON object. The two members of the JSON object are libs and code; both are optional.

The value of libs is either a string or an array of strings. Each string is the base name of a shared library that is part of the extension. The actual filename of the shared library depends on the platform; the SConstruct file that comes with the sample extension takes care of building a shared library with the right filename for the platform it is built on. This means that you can (and should) use the same SConstruct file to build the extension on all the platforms the extension supports.

The value of code is a filename of a KL source file (including the .kl extension), or an array of such filenames. All of the provided files are compiled and included into KL code wherever the require ExtensionName; statement is used to include the extension.

Fabric Engine detects an extension by finding a .fpm.json file somewhere in the extensions path. The extensions path includes all the directories listed in the FABRIC_EXTS_PATH environment variable, if present, as well as several predefined, platform-dependent directories that contain extensions that are included with the base install of Fabric Engine. Fabric Engine requires that the files referred to by the libs and code members in the extension manifest be in the same directory as the extension manifest file itself.

For more information about .fpm.json files, see The Extension Manifest File.

HelloWorld.kl¶

The HelloWorld.kl source file includes KL source code that is included whenever the require ExtensionName; statement includes the extension by name from other KL source. For instance, if a KL operator wants to use the functionality provided by the HelloWorld sample extension, it should include the statement require HelloWorld; at the top of the operator code. The contents of the HelloWorld.kl file are simple:

function String GetHelloWorldString() = "GetHelloWorldString";

This is a KL statement that refers to a function defined in the native code shared library. The key element here is the = "GetHelloWorldString" part: this tells KL that the function is called GetHelloWorldString in the native code shared library. If this is missing, there will be a linking problem when the library is loaded because by default KL functions have complex internal naming. The = "GetHelloWorldString" notation is simply telling KL the symbol name in the shared library.

Note that a KL source file can also include type definitions (eg. through the struct keyword), method definitions, operator definitions, and everything else available in KL. You can use the same = "..." notation to bind methods to code within the native code shared library.

HelloWorld.cpp¶

The file HelloWorld.cpp contains the C++ source code for the native code shared library. Its contents are:

#include "HelloWorld.h" // Automatically generated by kl2edk

using namespace Fabric::EDK;
IMPLEMENT_FABRIC_EDK_ENTRIES( HelloWorld )

FABRIC_EXT_EXPORT void GetHelloWorldString(
  KL::String::Result result
  )
{
  Fabric::EDK::log("Extension: Enter GetHelloWorldString");
  result = "Hello, world!";
  Fabric::EDK::log("Extension: Leave GetHelloWorldString");
}

The #include "HelloWorld.h" line includes the C++ header file with all the definitions for the the Fabric Engine EDK (Extension Development Kit).

The file HelloWorld.h is generated from HelloWorld.kl using the kl2edk utility; this action is automatically performed by the SConstruct build script. The kl2edk command generates C++ representations of all custom structure, object and interfaces types used in the KL source files, as well as function prototypes for the required C++ functions. For more information about kl2edk, see The kl2edk Utility.

The line using namespace Fabric::EDK; allows us to reference EDK functions and types without prefixing them with the EDK namespace; this is purely a notational convenience.

The IMPLEMENT_FABRIC_EDK_ENTRIES line is required to be present exactly once in ever extension. It takes a single parameter which must be the name of the extension. Its purpose is to implement internal functions that are used to load the extension.

注釈

There is a variant of IMPLEMENT_FABRIC_EDK_ENTRIES called IMPLEMENT_FABRIC_EDK_ENTRIES_WITH_SETUP_CALLBACK that takes a second parameter which is the address of a function to call at the end of the initialization process. It allows you to inject one-time setup code into extensions.

The remaining code is the definition of a function that is made available to KL. There are several important points:

  • Functions made available to KL must be prefixed with the FABRIC_EXT_EXPORT keyword. The purpose of the keyword is to ensure that the function has an unmangled symbol name and the right calling convention.
  • Note the use of the KL::String type for the first parameter, and note that this is the value that is returned from the function with the KL declaration function String GetHelloWorldString(). We will talk more about this in the section on calling conventions.

test.py¶

The test.py file is a Python source file that creates a simple Fabric Engine dependency graph and uses it to call the GetHelloWorldString function that is provided in the HelloWorld extension.

The first few lines of the file explicitly add the current directory to the FABRIC_EXTS_PATH environment variable:

import os
if 'FABRIC_EXTS_PATH' in os.environ:
  os.environ['FABRIC_EXTS_PATH'] = os.pathsep.join(['.', os.environ['FABRIC_EXTS_PATH']])
else:
  os.environ['FABRIC_EXTS_PATH'] = '.'

Usually a custom extension will be installed at a predefined location that is set in advance in the FABRIC_EXTS_PATH variable, but it is also possible to set the variable like this. The value of FABRIC_EXTS_PATH must be modified before the core client is created.

The next two lines create a Fabric Engine client. We communicate directly with the core of Fabric Engine as it is the easiest way to demonstrate the functionality of an extension:

import FabricEngine.Core as fabric
fabricClient = fabric.createClient();

The next several lines create an KL operator. The KL operator calls the GetHelloWorldString provided by the extension and stores the result in an io parameter that is later bound to a node member:

.. code-block:: python
opSource = [
“require HelloWorld;”, “”, “operator entry(io String string) {”, ” report(“KL: Enter entry”);”, ” string = GetHelloWorldString();”, ” report(“KL: GetHelloWorldString returned: ” + string);”, ” report(“KL: Leave entry”);”, “}”

] op = fabricClient.DG.createOperator(“op”) op.setSourceCode(“n”.join(opSource)) op.setEntryPoint(“entry”)

Note the first line require HelloWorld;: this statement is required to make available the declaration of the GetHelloWorldString function. The require statement essentially includes all of the KL source code that is provided in the extension; in this case, the code in HelloWorld.kl.

The next few lines create a node with a member of type String, binds the operator to the node and then evaluates the node. This causes the entry operator to run which in turn calls the GetHelloWorldString function in the extension:

b = fabricClient.DG.createBinding()
b.setOperator(op)
b.setParameterLayout(['self.string'])

node = fabricClient.DG.createNode("node")
node.addMember("string", "String")
node.bindings.append(b)
node.evaluate()

The next line verifies that the operator did in fact set the member to the right value:

print "Python got: " + node.getData("string", 0)

Finally, we close the client so that the script will exit:

fabricClient.close()

SConstruct¶

The SConstruct file is used to build the extension. The only lines of interest are the last ones:

fabricBuildEnv.Extension(
  'HelloWorld',
  ['HelloWorld.cpp', 'HelloWorld.kl']
  )

This line directs scons to build an extension named HelloWorld using the native code source file HelloWorld.cpp as well as the KL source code file HelloWorld.kl. The build commands cause the header file HelloWorld.h to be generated from the input KL source files using the kl2edk tool. The resulting HelloWorld.h is then included by HelloWorld.cpp.
